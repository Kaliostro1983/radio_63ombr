## 2. Доступ на інших ПК

Важливо розділяти **два різні шари**, які часто плутають:

### 2.1. Мережевий доступ (дотягнутися до застосунку) — Tailscale
- Кожен новий ПК додається у tailnet (авторизація пристрою).
- **ACL Tailscale**: позначити сервер тегом (напр. `tag:radio63-srv`), а операторам видати доступ **лише** до цього тегу на потрібному порту. Так новий ПК бачить сервіс, але не «всю мережу», і навпаки.
- Використовувати MagicDNS-ім'я (`ocheret-63.<tailnet>.ts.net`) замість сирого IP.

### 2.2. Ідентичність + автентифікація (хто це) — розвилка

**Варіант A (рекомендований): Tailscale-ідентичність через `tailscale serve` + тонкий реєстр користувачів у нас.**
- Застосунок ставимо за `tailscale serve` (HTTPS у межах tailnet). Serve інжектить перевірені заголовки ідентичності (`Tailscale-User-Login` = email/акаунт користувача) — які вже вміє читати `get_actor`.
- Кожен оператор заходить під **власним** Tailscale-акаунтом → отримуємо перевірену особу **без зберігання паролів у нас**. MFA/SSO — на боці Tailscale.
- У нас лишається лише таблиця `users`, ключ = Tailscale-логін, що задає **роль** + прапорець `enabled`.
- Плюси: не керуємо паролями; HTTPS «з коробки»; ідентичність криптографічно прив'язана до пристрою; збігається з наявним каркасом (`auth_context`).
- **Критична передумова безпеки:** прив'язати uvicorn до `127.0.0.1` (або лише інтерфейсу `tailscale0`), а не `0.0.0.0`. Інакше заголовок ідентичності можна підробити, звернувшись до порту 8000 в обхід проксі.

**Варіант B: власні акаунти (логін+пароль+сесія).**
- Своя таблиця `users` з **хешованими** паролями (bcrypt/argon2), логін-сторінка, підписані cookie-сесії (обов'язково полагодити secret_key).
- Плюси: незалежність від Tailscale-ідентичності; працює, навіть якщо кілька людей за одним пристроєм.
- Мінуси: ми відповідаємо за паролі (скидання, блокування після невдалих спроб, зберігання), більше коду й поверхні атаки.
- Обмеження процесу: паролі задає користувач самостійно; асистент паролі не вводить і не зберігає.

**Рекомендація:** **A як основа довіри + наш тонкий реєстр ролей**. Ідентичність — з Tailscale, права — наші. За потреби додати «аварійний» локальний адмін-логін на `127.0.0.1` для першого налаштування / доступу без Tailscale.

### 2.3. Алгоритм онбордингу нового оператора (варіант A)
1. Адмін запрошує Tailscale-акаунт людини у tailnet (або вхід через SSO-домен).
2. Пристрою призначається ACL-тег оператора; ACL дає доступ лише до `radio63-srv:443`.
3. Адмін створює рядок у `users` (`login` = Tailscale-email, `role` = `duty`, `enabled=1`).
4. Людина відкриває `https://ocheret-63.<tailnet>.ts.net/` → заголовок ідентичності → застосунок знаходить користувача → застосовує роль.
5. **Офбординг = два незалежні рубильники:** `enabled=0` у нас (миттєве блокування в застосунку) **і** видалення пристрою з tailnet (мережеве блокування).

### 2.4. Іменований логін — обов'язкова вимога (основа аудиту)
**Кожен акаунт працює під власним, персональним логіном.** Спільні/анонімні акаунти й «загальний доступ через мережу» — не допускаються, бо тоді неможливо встановити, хто саме вніс зміну.

- Логін (Tailscale-email у варіанті A або ім'я користувача у варіанті B) — це **канонічний ідентифікатор актора** й ключ, до якого прив'язується кожна зміна.
- Кожна дія, що змінює дані, атрибутується конкретному логіну — не IP, не пристрою, не «unknown». Наприклад: *«позивному X призначено статус „командир батальйону“»* має в журналі точний логін людини, час і значення до→після.
- Наслідок для автентифікації: сесія завжди має розвʼязуватися в **відомого, `enabled`** користувача. Якщо логіна встановити не вдалося (немає identity-заголовка / невідомий користувач) — доступ до мутацій закритий (не «unknown»-запис, а `401/403`).
- Наявний fallback у `get_actor` на IP/`"unknown"` для аудиту **недостатній** — його треба замінити на розвʼязання реального користувача (див. розділ 4).

### 2.5. Прив'язка ролі до комп'ютера (device-based role)

Роль призначається **робочому місцю (комп'ютеру)**, а не лише людині: конкретний ПК — це «місце чергового» чи «місце аналітика». Механізм із трьох частин.

**(1) Ідентифікатор пристрою — що технічно означає «цей комп'ютер».** Залежить від обраної основи автентифікації (§2.2):
- *Варіант A (Tailscale serve):* ідентичність пристрою від Tailscale (`Tailscale-Device-Name`/node-id) — стабільна, криптографічна, без жодного налаштування на ПК.
- *Варіант B / проміжний (без serve):* сервер при першому вході видає **пристрою постійний токен** — підписаний `device_key` (UUID), збережений як `httpOnly`-cookie. Прив'язаний до браузера/профілю на цьому ПК.

**(2) Центральний реєстр пристроїв — таблиця `devices`**, якою керує лише адмін:
`devices(device_key, label, role, enabled, first_seen, last_seen, created_by)`.
Приклад: `device_key=…, label="ПК чергового №1", role="operator", enabled=1`.

**(3) Потік реєстрації — самореєстрація + призначення адміном:**
1. Новий ПК відкриває панель → сервер не знає `device_key` → створює рядок зі статусом **`pending`** (жодних мутацій; максимум перегляд або блокуючий екран).
2. На ПК — екран: *«Цей комп'ютер не авторизовано. Код пристрою: `ABC-123`. Зверніться до адміністратора.»*
3. Адмін відкриває сторінку **«Пристрої»** → бачить pending-пристрій (код, IP, last-seen) → задає **роль** + мітку («ПК чергового №1») + `enabled=1`.
4. ПК оновлює сторінку → одразу отримує роль.
5. Адмін будь-коли **міняє роль** або **вимикає** пристрій (миттєво; це третій рубильник поряд із `users.enabled` і tailnet).

**Як роль резолвиться на кожен запит:** `device_key` (cookie/Tailscale) → пошук у `devices` → `role` + `enabled`. Невідомий/вимкнений пристрій → блок мутацій (403), лише перегляд або екран очікування.

**Співіснування з іменованим логіном (§2.4) — важливо не сплутати два різні поняття:**
- **Пристрій → РОЛЬ** (що дозволено на цьому робочому місці).
- **Логін → АКТОР** (хто саме зробив зміну — для журналу).

Це ортогональні речі. Рекомендація: **роль бере з пристрою, а актора для аудиту — з персонального логіну** (навіть легкого). Тоді «ПК чергового №1» має роль `operator`, але в журналі видно, що статус позивного змінив саме *Іванов*, а не абстрактний «комп'ютер».

> **Потрібне рішення:** чи вимагаємо персональний логін поверх ролі-пристрою (точний аудит «хто»), чи для виділеного односмінного робочого місця достатньо атрибуції на мітку пристрою («ПК чергового №1»)? Перше — точніше й відповідає §2.4; друге — простіше, але втрачає конкретну людину.

## 3. Ролі та доступний функціонал

### 3.1. Ролі (узгоджено — три ролі)
- **admin** (ви): усе — реєстр мереж (частоти/маски), статуси позивних, керування користувачами/ролями/пристроями, потоки/чати/налаштування, видалення, плюс усе, що можуть аналітик і оператор.
- **аналітик (черговий)**: повний аналітичний цикл — читання; **спільний** статус «переглянуто» (скоординована тріажна позначка між аналітиками); **збереження аналітичних висновків**; **пересилання повідомлень у месенджери** (Delta); коментарі та прив'язка позивних; імпорт/інжест; **орієнтири, палітри, втрати, словник** (рішення §7.3). **Не** редагує реєстр мереж, **не** призначає статуси позивних, **не** керує користувачами/пристроями/налаштуваннями.
- **оператор**: обмежений перегляд/тріаж — «аналітик мінус три можливості»:
  1. статус «переглянуто» — **лише локальний (per-user)**, щоб не впливати на чіпи в моніторингу аналітиків (див. 3.5);
  2. **БЕЗ** пересилання повідомлень у месенджери;
  3. **БЕЗ** збереження аналітичних висновків.
  Також грає роль спостерігача (окремого `viewer` не робимо — рішення §7.2).

**Принцип:** нові користувачі/пристрої за замовчуванням — найменші привілеї (`оператор`), підвищення — лише адмін.

### 3.2. Модель прав (capabilities, не розкидані перевірки ролей)
Замість `if role == "admin"` по коду — набір **capability** на роль, і кожен мутуючий ендпоінт вимагає конкретну capability. Чернетка переліку:

`network.edit`, `callsign.status.edit`, `conclusion.write`, `message.forward` (пересилання в месенджери/Delta), `message.comment`, `message.callsigns.edit`, `ingest.write`, `flow.manage`, `settings.manage`, `user.manage`, `push.manage`, `palette.manage`, `landmark.manage`, `casualty.manage`, `dictionary.manage`, `delete.*`, (`read` — неявно для будь-якого активного користувача).

Окремо: **статус «переглянуто»** — не звичайна capability, а модель даних (аналітики поділяють спільний, оператор має власний локальний; див. 3.5).

### 3.4. Забезпечення прав
- FastAPI-залежність `require(capability)`: визначає користувача з ідентичності → перевіряє `enabled` → перевіряє `роль→capability` → інакше `403`.
- **Кожен** мутуючий ендпоінт (усі ~90) отримує потрібну capability. GET-ендпоінти — лише «активний `enabled` користувач».
- UI: приховувати/вимикати недоступні ролі контроли — **але завжди дублювати перевіркою на сервері** (приховування у фронтенді — косметика, не захист). Зараз єдиний «захист» — це UI, і він захистом не є.

### 3.5. Статус «переглянуто» — per-user для оператора (вимога 1)
Зараз це **глобальна** колонка `messages.is_read` (0/1), спільна на всіх: `UPDATE messages SET is_read=1` у `app/routers/intercepts.py`, рендер `mon-thumb--unread` у `monitor.js`. Тобто будь-хто, хто відкрив повідомлення, гасить «непереглянуто» **для всіх**.

Вимога: перегляд оператора **не має впливати** на чіпи в моніторингу аналітиків. Дизайн:
- Аналітики й далі поділяють спільний `messages.is_read` (їхня скоординована тріажна позначка) — наявна поведінка не змінюється.
- Оператор отримує **власний локальний стан**: нова таблиця `message_read_state(user_id, message_id, read_at)`. Ендпоінт «позначити переглянутим» і вибірка списку для оператора читають/пишуть саме її, а глобальний `is_read` **не чіпають**.
- У відповіді моніторингу поле `is_read` обчислюється **за роллю**: для оператора — з його персонального стану, для аналітика — зі спільного.
- *(перспектива)* Модель дозволяє згодом зробити per-user для всіх ролей, якщо аналітики теж захочуть персональні позначки.

### 3.6. Точки пересилання в месенджери (мапа на права)
Пересилання розкидане по кількох місцях. Роль `оператор` не має жодного з них. Категорії:

**A. Інтерактивні (користувач тисне «Надіслати») — гейт `message.forward` (оператор ✗):**
- Кнопка **«Надіслати»** в модалці Delta на аналітичному висновку → `POST /api/push/send` (radio63 → бот → месенджер) + `POST /api/conclusions/{id}/mark-sended`. Це, по суті, єдина суто-ручна точка (`conclusions.js`, `push.py`).

**B. Конфігурація пересилання — гейт `settings.manage` (оператор ✗):**
- Налаштування Delta: чат/платформа + авто-надсилання по типах висновків.
- Вкладка «Імпорт/експорт»: чекбокс «активний» + вибір чату для надсилання помилок парсингу.

**C. Автоматичне (server-side) — НЕ дія ролі, керується конфігурацією/пайплайном:**
- Авто-надсилання висновків у Delta при інжесті — `_try_delta_auto_send` → `{bot}/api/push/send`.
- Авто-надсилання помилок парсингу в чат — `_try_import_error_send`.
- Фоновий воркер повторного надсилання — `start_delta_resend_worker`.

**Нюанс:** категорія C працює у фоні незалежно від того, хто залогінений (тримається на інжесті та налаштуваннях, а не на сесії). Оператор її не запускає й не конфігурує (немає ні `ingest.write`, ні `conclusion.write`, ні `settings.manage`). Тож із погляду оператора переслати в месенджер не можна ніяк: ні кнопкою (A), ні через налаштування (B).

## 4. Логування / аудит (хто що змінив)

**Мета:** для кожної дії, що змінює дані — хто / коли / що / до→після. **«Хто» = іменований логін** з розділу 2.4 (не IP, не пристрій, не «unknown»).

### 4.1. Дизайн — два шари
1. **Грубий (автоматичний, дешевий):** middleware пише в БД кожен не-GET запит: `actor`, `role`, `method`, `path`, `status`, `request_id`, `ip`, `ts`. Ловить усе, навіть якщо забули інструментувати конкретний ендпоінт. Замінює/доповнює нинішній ефемерний stderr-лог.
2. **Семантичний (точний):** на ключових мутаціях (висновок create/edit/delete, зміна поля мережі, зміна статусу позивного, коментар/прив'язка позивних, зміна користувача/ролі, налаштування) пишемо структурований рядок аудиту з `before`/`after` та людиночитним резюме. Саме це дає відповідь «хто і з чого на що змінив статус цього позивного».

### 4.2. Таблиця (чернетка)
`audit_log(id, ts_utc, request_id, actor, actor_role, ip, method, path, action, entity_type, entity_id, summary, before_json, after_json, status)`

- `request_id` (UUID на запит із middleware) пов'язує грубий і семантичний шари.
- Наявний `network_changes` — або лишаємо як є (вже виводиться в UI історії мережі) і паралельно пишемо в `audit_log`, або узагальнюємо. **Рішення потрібне.**

### 4.3. Перегляд і властивості
- Адмін-сторінка «Журнал дій» з фільтрами (актор, дата, тип сутності, дія) + історія на сутність («показати історію» на висновку/позивному/мережі).
- Аудит **append-only**: жодного UI на редагування/видалення рядків; перегляд — лише адмін.
- Ретенція: журнал росте — задати політику (напр. тримати рік у БД, старіше архівувати). За обсягами SQLite витримає.
- *(опційно)* хеш-ланцюг для tamper-evidence — надлишково для внутрішнього інструмента, лишаємо як можливість.

## 5. Інше, на що звернути увагу (доповнення)

### 5.1. Безпека (робити разом із рештою — це передумови)
- **Прив'язка uvicorn** до `127.0.0.1`/`tailscale0`, не `0.0.0.0` — інакше вся автентифікація обходиться будь-ким у LAN. Передумова для довіри до identity-заголовків.
- Замінити захардкоджений `secret_key="change_me_please"` на секрет з `.env` (з ротацією) — інакше cookie-сесії підробляються.
- **HTTPS** (через `tailscale serve`) — зараз чистий HTTP.
- **CSRF** для не-GET із cookie-сесіями (SameSite=lax допомагає частково; для варіанта B додати токен).
- Блокування після невдалих спроб входу (якщо варіант B).
- Секрети (`.env`, `data/panel-auth.json`) — не в git.

### 5.2. Конкурентність і цілісність (кілька операторів пишуть одночасно)
- Увімкнути **WAL** (`PRAGMA journal_mode=WAL`) + `busy_timeout` (напр. 5000 мс) + короткі транзакції. Зараз single-writer-припущення; кілька чергових одночасно → «database is locked».
- **Конфлікти правок (optimistic locking):** двоє редагують той самий висновок → last-write-wins губить дані. Додати `updated_at`/версію й попереджати про конфлікт (мінімум — для висновків).
- Якщо конкурентність зростатиме сильно — у перспективі розглянути Postgres (не зараз).

### 5.3. Операційне
- **Бекап перед розгортанням** і регулярні (є; перевірити, що `users`/`audit_log` потрапляють — так, копіюється весь файл).
- Це **міграція БД** (нові таблиці) — обкатати на копії; крок міграції зробити ідемпотентним.
- **Не заблокувати себе:** «break-glass» — локальний адмін на `127.0.0.1` або супер-адмін із `.env` на час розгортання.
- **Bootstrap адміна:** засіяти ваш адмін-акаунт **до** ввімкнення примусу прав (проблема курки і яйця).
- Тайм-аут сесії / «довіряти пристрою».
- Таймстемпи — UTC у БД, київський час в UI.

### 5.4. Процес/врядування
- Канонічний ідентифікатор користувача (Tailscale-email).
- Ролі видає лише адмін; за замовчуванням — `viewer`.
- Коротка інструкція для чергових: що можна / чого не можна.

## 6. Фази розгортання (детально, поетапно)

Кожна фаза — окремо розгортається й відкочується. Фази 0–1 **не потребують** рішень §7 (крім ретенції) і безпечні для продакшену вже зараз. Примус прав з'являється лише у Фазі 3. Легенда: 🟢 безпечно (не змінює поведінку) · 🟡 з'являється блокування · ⚙️ залежить від рішення §7.

### Фаза 0 — Безпекові передумови 🟢 (без зміни поведінки)
Мета: закрити базові діри й підготувати БД до конкурентності. Нічого не блокує, ролей ще немає.
- **0.1** `secret_key` сесій — читати з env `SESSION_SECRET` (генерувати випадковий; поки без ротації). Прибрати захардкоджений `"change_me_please"`.
- **0.2** `get_conn`: `PRAGMA journal_mode=WAL` + `busy_timeout=5000` + короткі транзакції — усуває «database is locked» при кількох записах.
- **0.3** Ручний бекап БД перед будь-якими міграціями наступних фаз; переконатися, що добовий бекап живий.
- **Готово коли:** застосунок працює як раніше; WAL увімкнено (перевірити `PRAGMA journal_mode`); секрет із env.
- **Відкат:** тривіальний (env-змінна + один PRAGMA).

### Фаза 1 — Фундамент даних + грубий аудит (лог-онлі) 🟢
Мета: створити схему й почати збирати журнал — **нічого не блокуючи**. Писати й далі можуть усі.
- **1.1** Міграції (ідемпотентні): таблиці `users`, `devices`, `audit_log`, `message_read_state` — створити, ще не використовувати для прав.
- **1.2** Bootstrap: засіяти ваш `admin`-акаунт і перший довірений пристрій (щоб не заблокувати себе далі).
- **1.3** Резолвер ідентичності — єдина залежність `current_actor(request)`: обчислює `login`, `role`, `device`, `enabled` (з Tailscale-заголовка / device-cookie / users). Поки **не блокує** — лише кладе результат у `request.state` і в лог.
- **1.4** Грубий аудит: middleware пише кожен не-GET запит у `audit_log` (`actor, role, method, path, status, request_id, ip, ts`). Режим спостереження.
- **Готово коли:** у `audit_log` накопичуються записи з реальним актором; жоден користувач нічого не втратив.
- **Відкат:** вимкнути middleware; таблиці лишаються порожні/незадіяні.

### Фаза 2 — Ідентичність і пристрої (доступ, ще без тонких ролей) 🟡⚙️
Мета: увімкнути реальну автентифікацію та реєстр пристроїв. Перше грубе блокування: «відомий і `enabled`» проти «невідомий». Ролі ще не розрізняються.
- **2.1** Механізм `device_key` (§2.5): cookie або Tailscale-ідентичність; сторінка **«Пристрої»** (адмін) + pending-екран для нового ПК.
- **2.2** Механізм логіну ⚙️ (**рішення §7.1**): або `tailscale serve` з identity-заголовками, або власна логін-сторінка.
- **2.3** Гейт мутацій: писати може лише відомий `enabled` користувач/пристрій; невідомий → `403`/екран очікування. Читання — за рішенням (відкрите чи теж гейтимо).
- **2.4** Прив'язка uvicorn за проксі/інтерфейсом (тепер безпечно, бо identity налаштовано) + HTTPS через serve.
- **Готово коли:** новий ПК проходить реєстрацію → адмін вмикає → отримує доступ; невідомі заблоковані; у журналі — іменований актор.
- **Відкат:** прапорець «не примушувати ідентичність» повертає відкритий режим.

### Фаза 3 — Ролі та примус прав 🟡⚙️
Мета: увімкнути розмежування за ролями. Робимо **поетапно, ендпоінт за ендпоінтом**, щоб не зламати робочі процеси.
- **3.1** Залежність `require(capability)` + мапа capability→роль (§3.2–3.3).
- **3.2** Інструментувати мутації, починаючи з **чутливих**: `network.edit`, `callsign.status.edit`, `user.manage`, `device.manage`. Далі — `conclusion.write`, `message.forward`, `ingest.write`, `settings.manage`, і решта.
- **3.3** Оператор (вимоги §3.1): локальний `message_read_state` (per-user «переглянуто») + закриті `message.forward` і `conclusion.write`.
- **3.4** UI за роллю: приховати/вимкнути недоступні контроли (косметика поверх серверного примусу).
- **3.5** ⚙️ Врахувати рішення §7.2 (viewer), §7.3 (права аналітика по краях).
- **Готово коли:** кожна роль бачить/може рівно те, що в матриці; спроба «через голову» повертає `403`.
- **Відкат:** покапабіліті — прибрати `require(...)` з конкретного ендпоінта.

### Фаза 4 — Семантичний аудит + журнал дій ⚙️
Мета: журнал «хто/що/до→після» і зручний перегляд.
- **4.1** before/after на ключових мутаціях: висновки (create/edit/delete), статуси позивних, поля мереж, користувачі/пристрої, налаштування.
- **4.2** Адмін-сторінка **«Журнал дій»**: фільтри (актор, дата, тип сутності, дія) + «історія» на сутність.
- **4.3** ⚙️ Рішення §7.4 (`network_changes`: лишити / злити в `audit_log`).
- **Готово коли:** на будь-яку зміну видно хто/коли/що/до→після; є сторінка перегляду.

### Фаза 5 — Цілісність і шліфування ⚙️
- **5.1** Конфлікти правок (optimistic locking) для висновків: `updated_at`/версія + попередження про конфлікт.
- **5.2** ⚙️ Ретенція/архів аудиту (**рішення §7.5**).
- **5.3** Тайм-аути сесій, «довіряти пристрою», процедура офбордингу, коротка інструкція ролям.

### Залежності фаз від рішень §7
| Фаза | Потрібні рішення §7 |
|---|---|
| 0, 1 | — (можна починати зараз) |
| 2 | §7.1 (автентифікація), §7.6 (роль-пристрій vs логін) |
| 3 | §7.2 (viewer), §7.3 (права аналітика) |
| 4 | §7.4 (network_changes) |
| 5 | §7.5 (ретенція) |

## 7. Узгоджені рішення (13.07.2026) ✅

1. **Основа автентифікації** → **варіант A+**: Tailscale-ідентичність через `tailscale serve` (HTTPS, без паролів у нас) + **аварійний локальний адмін** на `127.0.0.1` для розгортання/збоїв.
2. **Набір ролей** → **три ролі**: `admin` / `аналітик` / `оператор`. Окремого `viewer` не робимо — його нішу покриває «оператор» (тільки з локальним «переглянуто»). Окремого «реєстратора частот» не потрібно.
3. **Права аналітика по краях** → **усе аналітику**: `landmark.manage`, `palette.manage`, `casualty.manage`, `dictionary.manage` — усі ці дії доступні аналітику (не лише адміну).
4. **`network_changes`** → **лишити окремо** й паралельно писати в `audit_log` (уніфікація — можливе пізніше, без ризику для наявної історії мереж в UI).
5. **Ретенція аудиту** → **12 місяців онлайн**, старіше — в архів.
6. **Роль-пристрій vs логін (§2.5)** → **роль з пристрою + актор з персонального логіну**: робоче місце задає роль, а в журналі фіксується конкретна людина (відповідає §2.4).
