# Контракт бота ↔ FastAPI: `POST /api/ingest/whatsapp`

Джерело правди в коді: `app/routers/ingest.py` (HTTP), `app/services/ingest_service.py` → `process_whatsapp_payload`.  
Поле `flow_id` / `flow_name` у клієнта (`index.cjs` тощо) у бекенді **не читаються** — їх можна надсилати для власних логів бота; FastAPI приймає довільні додаткові ключі JSON і ігнорує невідомі.

---

## Запит

- **Метод / шлях:** `POST /api/ingest/whatsapp`
- **Заголовок:** `Content-Type: application/json`
- **Тіло:** один JSON-об’єкт (об’єкт на верхньому рівні, не масив).

### Поля тіла POST (використання сервером)

| Поле | Тип | Обов’язкове | Опис |
|------|-----|-------------|------|
| `chat_id` | string | **так** | Зовнішній ідентифікатор чату на платформі; зберігається як `source_chat_id` у `ingest_messages`. |
| `message_id` | string | **так** | Унікальний ідентифікатор повідомлення на платформі; разом з `platform` забезпечує унікальність рядка інжесту. |
| `text` | string | **так** | Сирий текст перехоплення. Порожній рядок не проходить перевірку (разом з відсутніми `chat_id` / `message_id` повертається помилка). |
| `platform` | string | ні | Джерело; за замовчуванням `"whatsapp"`. Також використовується логікою (наприклад `"xlsx"` у імпорті). |
| `chat_name` | string \| null | ні | Назва чату для збереження в метаданих інжесту. |
| `published_at_platform` | string \| null | ні | Час події на стороні платформи (зазвичай ISO); для розрахунку `delay_sec` порівняно з часом у тексті перехоплення. |
| `allow_send` | boolean | ні | За замовчуванням `false`. Якщо `true` і `text` непорожній, у **відповіді** додається елемент у масиві `actions` (див. нижче). |

### Типові додаткові поля з бота (не обробляються FastAPI)

У договорі з іншим репозиторієм (наприклад `index.cjs`) можуть бути, зокрема:

- `flow_id`, `flow_name` — для маршрутизації / логів на боці бота;
- інші службові поля.

Вони **не згадані** у `process_whatsapp_payload` і на поведінку сервера не впливають.

## Відповідь

- **Успішна обробка з помилкою валідації на вході** (немає обов’язкових полів): HTTP **400**, тіло на кшталт:

```json
{
  "ok": false,
  "error": "chat_id, message_id, text are required"
}
```

У цьому випадку поля **`actions` немає**.

- **Усі інші відповіді** з `process_whatsapp_payload` (у т.ч. `ok: true` з `skipped`, `duplicate`, вставка повідомлення тощо) містать масив **`actions`**.

### Масив `actions`

- Тип: масив об’єктів.
- Мінімальний контракт для бота: кожен елемент має поле **`type`** (рядок).
- Зараз сервер формує лише один тип:

| `type` | Інші поля | Коли з’являється |
|--------|-----------|------------------|
| `send_message` | **`text`** (рядок) | Якщо в запиті `allow_send === true` і `text` непорожній. Текст у відповіді — той самий, що в тілі запиту (`text`). |

Приклад фрагмента успішної відповіді:

```json
{
  "ok": true,
  "ingest_id": 123,
  "actions": [
    { "type": "send_message", "text": "…оригінальний текст перехоплення…" }
  ]
}
```

Якщо `allow_send` false або `text` порожній, **`actions` зазвичай `[]`**.

Очікувана поведінка бота: пройтися по `actions`, для `type === "send_message"` виконати відправку повідомлення з полем `text` (наприклад у той самий чат / за правилами вашого `flow_id`).

### Інші поля відповіді (довідково)

Залежно від гілки парсингу сервер може повернути, зокрема: `ingest_id`, `message_row_id`, `skipped`, `reason`, `duplicate`, `parsed`, `peleng_batch_id`, `details` тощо — для узгодження з UI/логами; для мінімального бота достатньо `ok` / `error` та `actions`.
