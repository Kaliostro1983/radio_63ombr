Відповіді на поширені запитання.
Не вдається скопіювати зображення карти — з'являється повідомлення «Копіювання недоступне — файл завантажено»

Кнопка «📸 Копіювати карту» використовує API браузера navigator.clipboard.write(), який дозволений тільки у захищеному контексті — тобто на https:// або localhost. При доступі до сервісу через звичайний http:// із мережі браузер блокує цей виклик, і замість копіювання файл автоматично скачується.

Рішення — прапор Chrome (виконується один раз на кожному комп'ютері):

  1. Відкрий Chrome і в адресному рядку введи:
    chrome://flags/#unsafely-treat-insecure-origin-as-secure
  2. У текстовому полі «Insecure origins treated as secure» введи адресу сервісу, наприклад:
    http://100.94.179.49:8000
    (підстав свою IP-адресу, якщо відрізняється)
  3. У дропдауні праворуч від поля вибери Enabled.
  4. Натисни кнопку Relaunch — Chrome перезапуститься.

Після цього Chrome вважатиме даний HTTP-адрес «безпечним» і копіювання зображень у буфер обміну запрацює в штатному режимі.

Якщо в майбутньому на сервері буде налаштовано HTTPS — цей прапор можна прибрати, і копіювання заробить без додаткових налаштувань браузера.
Кнопки «Copy» / «Скопіювати» не копіюють текст при віддаленому доступі через HTTP

Копіювання тексту через navigator.clipboard.writeText() також вимагає захищеного контексту. Однак для тексту в сервісі реалізовано автоматичний фолбек через document.execCommand('copy'), який працює навіть на звичайному HTTP. Тому кнопки копіювання тексту мають працювати коректно.

Якщо все ж виникає помилка «Не вдалося скопіювати» для тексту — застосуй той самий прапор Chrome, описаний у питанні вище.

Сервіс не запускається, в логах — помилка «database disk image is malformed»

SQLite-файл може пошкодитися через некоректне вимкнення машини (відключення живлення, примусове завершення процесу). Сервіс має вбудований механізм відновлення: при старті він перевіряє цілісність БД командою PRAGMA quick_check і намагається відновити файл через VACUUM INTO.

Якщо автовідновлення не спрацювало:

  1. Зупини сервіс.
  2. Скопіюй справний файл БД з іншої машини (або з резервної копії з папки backups/) і заміни ним пошкоджений.
  3. Запусти сервіс повторно.

Резервні копії створюються автоматично при кожному запуску сервісу і зберігаються в папці backups/ поруч із основним файлом БД.
